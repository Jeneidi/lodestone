1. How the Whole Project Fits Together

Before reading any code, understand the flow of data from a user's question to the final answer. Everything in this codebase exists to serve this pipeline:

User types: "Who was Nikola Tesla?" │ ▼ engine.ask("Who was Nikola Tesla?") │ ├─► BM25Retriever.search() ← keyword match (finds "Tesla", "Nikola") │ │ ├─► DenseRetriever.search() ← meaning match (finds semantically similar chunks) │ │ ▼ HybridRetriever fuses both lists (RRF or weighted) │ ▼ CrossEncoderReranker.rerank() ← re-reads each (question, chunk) pair deeply │ ▼ ExtractiveAnswerer.answer() ← picks top chunk's text as the answer │ ▼ Answer returned to dashboard/API/CLI
The key mental model: data shapes A Document is a whole Wikipedia article. A Chunk is a piece of it (~200 words). A ScoredChunk is a Chunk that a retriever ranked with a score. An Answer is the final output. These four objects flow through every file.

2. schemas.py — The Data Contracts

This file is the shared language of the whole project. Every other file imports from here. If you change a field name here, you break everything. Read this first.

src/lodestone/schemas.py
from __future__ import annotations from pydantic import BaseModel, Field
Lines 1–2 — imports from __future__ import annotations — tells Python to treat all type hints as text strings, not real objects. This lets you write list[str] without Python trying to evaluate it at startup. Always put this at the top of files with type hints. Why: without this, on Python 3.9 and below, list[Chunk] would crash because Python would try to evaluate "list of Chunk" before Chunk is defined.
Library: __future__ is Python stdlib. pydantic is a third-party data validation library (pip install pydantic).
class Document(BaseModel): doc_id: str title: str = "" text: str source: str = "" metadata: dict = Field(default_factory=dict)
class Document(BaseModel) — pydantic model class Document(BaseModel) creates a data class that validates its own fields. When you do Document(doc_id="abc", text="hello") pydantic checks that doc_id is actually a string, text is a string, etc. If you pass the wrong type it raises an error immediately. Why: without pydantic, you'd pass raw dicts everywhere and typos like {"dco_id": "abc"} would silently fail. Pydantic catches bugs at the boundary.
doc_id: str A field with no default — it is REQUIRED. If you try to create a Document without doc_id, pydantic raises a ValidationError. Why: every Document needs a unique ID so retrievers can match chunks back to their parent document when computing retrieval metrics.
title: str = "" A field WITH a default — it is OPTIONAL. If you don't provide title, it becomes an empty string automatically. Why: not all corpora have titles (some are raw text dumps), so making it optional keeps the model flexible.
metadata: dict = Field(default_factory=dict) Field(default_factory=dict) means "call dict() to create the default". This is different from metadata: dict = {}. Why this matters: in Python, = {} as a default is dangerous — ALL instances would share the SAME dictionary object. default_factory=dict creates a fresh empty dict for each instance. This is a classic Python gotcha.
class Chunk(BaseModel): chunk_id: str doc_id: str text: str index: int = 0 metadata: dict = Field(default_factory=dict)
Chunk — a piece of a Document A Document gets split into many Chunks. doc_id links back to the parent Document. index is its position within that document (0 = first chunk, 1 = second, etc.). chunk_id is globally unique (built as "{doc_id}::fixed::{index}"). Why: you need chunks because LLMs and retrievers work best on short passages (~200 words), not 2000-word Wikipedia articles. The chunk_id lets you deduplicate results and trace where each result came from.
class ScoredChunk(BaseModel): chunk: Chunk score: float retriever: str = ""
ScoredChunk — a Chunk with a relevance score attached chunk: Chunk means this field IS a Chunk object (nested model). score: float is the relevance score from whichever retriever found it. retriever: str tracks which retriever gave this score (e.g. "bm25", "dense", "hybrid_rrf"). Why: after retrieval, you need to know both WHICH chunk was relevant AND HOW relevant it was. The score is used for sorting and merging. The retriever name is shown in the dashboard and eval reports.
class QAExample(BaseModel): qid: str question: str answer: str relevant_doc_ids: list[str] = Field(default_factory=list)
QAExample — one question + its ground-truth answer Used ONLY by the eval harness, not by the live system. relevant_doc_ids is a list of doc_ids that are the correct answer sources. The eval code checks: "did the retriever find any chunk from these doc_ids?" Why: you need ground truth to measure retrieval quality. Without QAExamples, you can't calculate Recall@5 or nDCG — you'd have no way to know if the system found the right thing.
class Answer(BaseModel): text: str supporting_chunks: list[ScoredChunk] = Field(default_factory=list) generator: str = "" faithfulness: float | None = None latency_ms: float = 0.0
Answer — the final output returned to users text is the answer string. supporting_chunks are the passages used as evidence. generator is either "extractive" or "claude". faithfulness: float | None means it can be a float OR None — the | is Python's union type syntax. It's None when faithfulness scoring is off. Why: bundling all this into one object means every layer (API, CLI, dashboard) gets everything it needs from one return value instead of multiple function calls.
class RetrievalRunResult(BaseModel): qid: str retrieved: list[ScoredChunk] latency_ms: float = 0.0
RetrievalRunResult — eval harness output per question Records what the retriever returned for one question (qid links to a QAExample). Used by evals/aggregate.py to compute metrics. Why: the eval harness runs hundreds of questions, so it needs a structured format to store all results before computing aggregate statistics. Separating retrieval results from metric computation means you can re-run metrics without re-running retrieval.

3. config.py — Settings and Environment Variables

This file controls every tunable parameter in the system without touching code. Change a model name or threshold by setting an environment variable.

src/lodestone/config.py
import functools from pydantic import Field from pydantic_settings import BaseSettings, SettingsConfigDict
imports functools gives us lru_cache to cache the settings singleton. pydantic_settings is a separate package from pydantic that reads values from environment variables and .env files automatically. Library: functools is stdlib. pydantic-settings is a third-party package.
class Settings(BaseSettings): model_config = SettingsConfigDict( env_prefix="LODESTONE_", env_file=".env", env_file_encoding="utf-8", extra="ignore", )
class Settings(BaseSettings) — env-driven config BaseSettings is like BaseModel but reads field values from environment variables automatically. env_prefix="LODESTONE_" means the env var for a field called top_k is LODESTONE_TOP_K. env_file=".env" means it also reads a .env file if present. extra="ignore" means unknown env vars are silently ignored (no crash). Why: hardcoding model names and thresholds into source code makes experimentation painful. With this pattern, you can change any setting by setting an env var — no code change required. The prefix prevents collisions with other apps' env vars.
embedding_model_name: str = Field( default="sentence-transformers/all-MiniLM-L6-v2", description="HuggingFace model identifier for dense retrieval.", )
Field with description Field(default=..., description=...) sets both the default value and human-readable documentation. To override: export LODESTONE_EMBEDDING_MODEL_NAME="sentence-transformers/all-mpnet-base-v2". Why: the description shows up in IDE autocompletion and makes the .env.example file self-documenting. The default is MiniLM because it's fast, free, and good enough for CPU inference.
rrf_k: int = Field(default=60, ge=1, description="RRF smoothing constant.") hybrid_alpha: float = Field(default=0.5, ge=0.0, le=1.0, ...)
ge= and le= — validation constraints ge=1 means "greater than or equal to 1". le=1.0 means "less than or equal to 1.0". Pydantic enforces these at runtime — if you set LODESTONE_HYBRID_ALPHA=5.0 it raises a ValidationError immediately. Why: these constraints catch configuration mistakes before they corrupt results silently. alpha=5.0 would produce nonsensical weighted scores; catching it early is better than debugging wrong eval numbers later.
anthropic_api_key: str | None = Field(default=None, ...)
Optional secret field str | None means this field can be a string OR None. Default is None, meaning Claude generation is unavailable unless you explicitly set the key. The | syntax is Python 3.10+ union types. Why: the whole system works without an API key (using extractive answers). Making the key optional means zero-cost local operation by default, with Claude as an upgrade path.
@functools.lru_cache(maxsize=1) def get_settings() -> Settings: return Settings()
@functools.lru_cache(maxsize=1) — the singleton pattern @functools.lru_cache is a decorator that caches the return value of a function. maxsize=1 means it caches exactly one call result. So get_settings() reads the environment and builds the Settings object ONCE, then returns the same object on every subsequent call. Why: creating a Settings object reads from env vars and disk. Doing that 500 times during a search query would be wasteful. The cache means it's done once at startup. The maxsize=1 ensures there's always exactly one Settings instance in the whole process — a singleton.

4. retrieval/tokenize.py — Cleaning Text Into Words

BM25 is a keyword search algorithm. Before you can match keywords, you need to convert messy raw text into clean, comparable word tokens. That's this file's entire job.

src/lodestone/retrieval/tokenize.py
import re import unicodedata STOPWORDS: frozenset[str] = frozenset({"a", "an", "the", "and", "or", ...})
re and unicodedata — stdlib text tools re = regular expressions. Used to find sequences of word characters. unicodedata = Unicode text processing. Used to normalize accented characters ("é" → "e"). Library: both are Python stdlib — no install needed.
STOPWORDS: frozenset[str] A frozenset is an immutable set — like a set but you can't add or remove items after creation. Checking "the" in STOPWORDS is O(1) (instant) because sets use hash lookup, not linear scan. Why frozenset instead of a list: membership check word in frozenset is O(1). With a list it would be O(n) — slow when called millions of times during indexing. Why these specific words: stop words ("the", "a", "is") appear in nearly every document so they carry zero discriminating information for keyword search.
_WORD_RE = re.compile(r"\w+")
re.compile() — pre-compiled regex re.compile(r"\w+") compiles a regex pattern once at module load time. \w+ means "one or more word characters" (letters, digits, underscore). The r"" prefix means raw string — backslashes are literal, not escape characters. Why compile instead of re.findall(r"\w+", text) inline: compiling once is faster when the same pattern is used thousands of times (during indexing every chunk). The compiled object stores the pattern's bytecode.
def normalize(text: str) -> str: return unicodedata.normalize("NFKD", text).lower()
normalize() — unicode normalization + lowercasing unicodedata.normalize("NFKD", text) decomposes characters like "é" into "e" + combining accent mark. .lower() lowercases everything. Result: "Nikola" → "nikola", "café" → "cafe" (approximately). Why: BM25 does exact string matching. "Tesla" and "tesla" should match. "résumé" and "resume" should match. Without normalization, capitalization and accents would prevent matches.
def word_tokenize(text: str) -> list[str]: return _WORD_RE.findall(text)
word_tokenize() — split text into word tokens _WORD_RE.findall(text) returns a list of all non-overlapping matches of \w+ in text. "Hello, world!" → ["Hello", "world"]. Punctuation and spaces are automatically discarded (they don't match \w+). Why not just text.split(): split() keeps punctuation attached ("world!" stays "world!"). The regex cleanly strips all non-word characters.
def remove_stopwords(tokens: list[str]) -> list[str]: return [t for t in tokens if t not in STOPWORDS]
List comprehension with filter condition [t for t in tokens if t not in STOPWORDS] — this is a list comprehension. It means: "go through every token t in the list; keep it only if it's NOT in STOPWORDS". Returns a new list (does not modify the original). Why: "who is nikola tesla" after stopword removal becomes ["nikola", "tesla"]. Only content words remain. This improves BM25 quality because "is" appearing in every document would dilute scores if kept.
def tokenize(text: str, remove_stops: bool = True, stem: bool = False) -> list[str]: tokens = word_tokenize(normalize(text)) if remove_stops: tokens = remove_stopwords(tokens) if stem: tokens = [_porter_lite_stem(t) for t in tokens] return tokens
tokenize() — the complete pipeline in one function This chains the three steps: normalize → word_tokenize → (optionally) remove_stopwords → (optionally) stem. The caller decides what they need. BM25 calls it with remove_stops=True, stem=False. Note the nested call: word_tokenize(normalize(text)) — normalize runs first, its output feeds into word_tokenize. Why this design: each step is separately testable. You can test normalize() alone, word_tokenize() alone, etc. The composed tokenize() just wires them together with flags for optional steps.
def _porter_lite_stem(token: str) -> str: if len(token) <= 3: return token if token.endswith("ness") and len(token) - 4 >= 4: return token[:-4] if token.endswith("ing") and len(token) - 3 >= 3: stem = token[:-3] if len(stem) >= 2 and stem[-1] == stem[-2] and stem[-1] not in "aeiou": return stem[:-1] # runn → run ...
_porter_lite_stem() — suffix stripping Strips common English suffixes so "running" and "run" match as the same word. token[:-4] means "all characters except the last 4" (removes "-ness"). The doubled-consonant check (stem[-1] == stem[-2]) handles "running" → "runn" → "run". The underscore prefix means this is a private function not meant to be called from outside this module. Why not use NLTK: NLTK requires a download of data files and is heavy. This mini-stemmer handles 90% of cases with zero dependencies. It's documented as imperfect — that's intentional and honest.

5. retrieval/base.py — The Retriever Contract

This file defines a rule: any class that wants to be a retriever MUST have index() and search(). It enforces this at the Python language level.

src/lodestone/retrieval/base.py
from abc import ABC, abstractmethod from lodestone.schemas import Chunk, ScoredChunk class Retriever(ABC): name: str = "retriever" @abstractmethod def index(self, chunks: list[Chunk]) -> None: ... @abstractmethod def search(self, query: str, k: int = 10) -> list[ScoredChunk]: ...
ABC and @abstractmethod — the abstract base class pattern ABC means "Abstract Base Class". A class that inherits from ABC cannot be instantiated directly — you must subclass it and implement all @abstractmethod methods. If you try to create a subclass without implementing index() or search(), Python raises a TypeError when you try to instantiate it. Why: the HybridRetriever, eval harness, and engine all accept any Retriever. Without this contract, nothing stops someone from writing a retriever that has query() instead of search(), which would crash at runtime. The ABC catches that at class definition time.
Library: abc is Python stdlib.
name: str = "retriever" — class-level attribute Every concrete subclass overrides this with its own name (e.g., name = "bm25"). This name gets stored in ScoredChunk.retriever so you can see which retriever produced each result.
... (ellipsis) as a method body The ... (called Ellipsis) is valid Python — it means "no body, to be filled in by subclasses". It's equivalent to pass here but more idiomatic for abstract methods.

6. retrieval/bm25.py — Keyword Search From Scratch

This is the most algorithmic file in the project. BM25 is a 1994 algorithm that still powers much of Google's early ranking and almost every search engine's keyword layer.

The BM25 formula you need to understand: score(chunk, query) = sum over each query word of: IDF(word) × TF_norm(word, chunk)

IDF(word) = how rare is this word across all chunks? Rare words score higher.
TF_norm(word, chunk) = how often does this word appear in this chunk, normalized by chunk length?

Together: a chunk scores high when it contains rare query words, frequently, in a shorter text.
src/lodestone/retrieval/bm25.py
from collections import Counter, defaultdict import math from lodestone.retrieval.base import Retriever from lodestone.retrieval.tokenize import tokenize from lodestone.schemas import Chunk, ScoredChunk
Counter and defaultdict — smart dictionary types Counter(["a","b","a","c","a"]) → {"a": 3, "b": 1, "c": 1}. Perfect for term frequency. defaultdict(list) is a dict that automatically creates a [] for any missing key instead of raising a KeyError. Library: collections is Python stdlib.
class BM25Retriever(Retriever): name: str = "bm25" def __init__(self, k1: float = 1.5, b: float = 0.75) -> None: self.k1 = k1 self.b = b self._chunks: list[Chunk] = [] self._doc_lengths: list[int] = [] self._avgdl: float = 0.0 self._inverted_index: dict[str, list[tuple[int, int]]] = defaultdict(list) self._idf: dict[str, float] = {} self._indexed: bool = False
__init__ — all internal state starts empty Everything starting with self._ is private state that index() will fill in later. _inverted_index is the core structure: a dict mapping each word to a list of (chunk_index, how_many_times) pairs. _doc_lengths[i] holds the token count of chunk number i. _avgdl is the average of all those lengths. Why k1=1.5 and b=0.75: these are the BM25 standard defaults from the original paper, extensively validated across many corpora. k1 controls how much each additional occurrence of a word increases the score (higher = more credit). b controls how much long documents are penalized (0=none, 1=full).
def index(self, chunks: list[Chunk]) -> None: if not chunks: raise ValueError("BM25Retriever.index() received an empty chunk list.") self._chunks = list(chunks) n = len(chunks) forward: list[Counter[str]] = [] for chunk in chunks: tf: Counter[str] = Counter(tokenize(chunk.text, remove_stops=True, stem=False)) forward.append(tf) self._doc_lengths.append(sum(tf.values()))
Building the "forward index" — term frequencies per chunk Counter(tokenize(chunk.text)) — tokenize splits the chunk into clean words, Counter counts how often each word appears. sum(tf.values()) adds up all the counts to get the total word count of the chunk. This builds the "forward index" — for each chunk, what words and how many times. Why store doc_lengths: BM25 penalizes longer documents. A chunk with 500 words that mentions "tesla" once should score lower than a 50-word chunk that mentions it once. The length normalizes for this.
self._avgdl = sum(self._doc_lengths) / n if n > 0 else 1.0 self._inverted_index = defaultdict(list) df: Counter[str] = Counter() for chunk_idx, tf in enumerate(forward): for term, freq in tf.items(): self._inverted_index[term].append((chunk_idx, freq)) df[term] += 1
Building the inverted index — from forward to reverse lookup enumerate(forward) gives (0, counter_for_chunk_0), (1, counter_for_chunk_1), etc. For each word in each chunk, we add (chunk_index, frequency) to the inverted index. df[term] += 1 counts how many chunks contain this term (document frequency). This is the key data structure: instead of "for each chunk, what words does it contain?" it becomes "for each word, which chunks contain it?" Now when searching, you look up the query word and instantly get the list of relevant chunks — you never scan all chunks.
Example result: _inverted_index["tesla"] = [(3, 2), (17, 5), (42, 1)] means chunk 3 has "tesla" twice, chunk 17 has it five times, chunk 42 has it once.
self._idf = {} for term, doc_freq in df.items(): self._idf[term] = math.log((n - doc_freq + 0.5) / (doc_freq + 0.5) + 1.0)
IDF calculation — how rare is each word? n = total chunks. doc_freq = how many chunks contain this term. The formula log((n - df + 0.5) / (df + 0.5) + 1) outputs a high number for rare words (df is small → numerator large) and a low number for common words (df is large → numerator small). The + 1 outside the log prevents the IDF from going negative when df = n (word appears in every chunk). Why precompute: you need IDF for every search query. Computing it once during indexing means each search query just looks it up in self._idf — O(1) lookup vs recomputing over the whole corpus on every query.
def search(self, query: str, k: int = 10) -> list[ScoredChunk]: if not self._indexed: raise RuntimeError("BM25Retriever.search() called before index().") query_terms = tokenize(query, remove_stops=True, stem=False) if not query_terms: return [] scores: dict[int, float] = {} for term in query_terms: if term not in self._inverted_index: continue idf = self._idf.get(term, 0.0) for chunk_idx, tf in self._inverted_index[term]: dl = self._doc_lengths[chunk_idx] tf_norm = tf * (self.k1 + 1.0) / ( tf + self.k1 * (1.0 - self.b + self.b * dl / self._avgdl) ) scores[chunk_idx] = scores.get(chunk_idx, 0.0) + idf * tf_norm
The BM25 scoring loop For each query word: look it up in the inverted index → get the list of chunks that contain it → for each such chunk, compute the BM25 term score and add it to that chunk's running total. scores.get(chunk_idx, 0.0) returns 0.0 if that chunk hasn't been scored yet (default for missing keys).
The tf_norm formula: tf * (k1+1) / (tf + k1*(1 - b + b*dl/avgdl)). Breaking it down: the numerator grows with term frequency. The denominator also grows with tf but slower (saturation — tenth occurrence matters less than first). The b * dl/avgdl part penalizes longer chunks. Why not just count occurrences: a chunk that says "Tesla Tesla Tesla Tesla Tesla" 50 times shouldn't score 50× better than one that mentions it once with important context. The saturation formula handles this.
ranked = sorted( scores.items(), key=lambda item: (-item[1], self._chunks[item[0]].chunk_id), ) results: list[ScoredChunk] = [] for chunk_idx, score in ranked[:k]: results.append(ScoredChunk( chunk=self._chunks[chunk_idx], score=score, retriever=self.name, )) return results
Sorting and building the result list scores.items() gives [(chunk_idx, score), ...]. key=lambda item: (-item[1], self._chunks[item[0]].chunk_id) — the lambda is an anonymous function. It sorts by -score first (negative so highest scores sort first) and by chunk_id second (for tie-breaking). ranked[:k] takes the top k. Why sort descending: most relevant first. Why tie-break by chunk_id: without a tiebreaker, two chunks with the same score could appear in random order — making tests flaky and results non-reproducible. The chunk_id tie-break guarantees identical output every time.

7. retrieval/dense.py — Meaning-Based Search

Where BM25 matches exact words, dense retrieval matches meaning. It converts text into numerical vectors (embeddings) where similar texts have similar vectors. Finding relevant chunks = finding vectors close to the query vector.

The key idea:
"Who invented alternating current?" and "Tesla developed AC electricity" have no words in common — BM25 scores them 0. But their embedding vectors are close in 384-dimensional space, so dense retrieval scores them high. That's why combining both is better than either alone.
src/lodestone/retrieval/dense.py
from collections.abc import Callable import numpy as np from lodestone.retrieval.base import Retriever from lodestone.schemas import Chunk, ScoredChunk
Callable and numpy Callable is a type hint for "something you can call like a function". Used to type-hint the optional encoder parameter. numpy (imported as np by convention) is the numerical computing library that makes matrix math fast. Library: collections.abc is stdlib. numpy is third-party.
def _l2_normalize(matrix: np.ndarray) -> np.ndarray: norms = np.linalg.norm(matrix, axis=1, keepdims=True) norms = np.where(norms == 0.0, 1.0, norms) matrix /= norms return matrix
L2 normalization — making all vectors unit length np.linalg.norm(matrix, axis=1) computes the length (L2 norm) of each row vector. axis=1 means "compute across columns for each row". keepdims=True keeps the result as shape (n,1) so it can broadcast-divide into the (n,d) matrix. np.where(norms==0.0, 1.0, norms) replaces zeros with 1 (avoid division by zero). matrix /= norms divides every element by its row's norm in-place. Why normalize: when all vectors have length 1, cosine similarity (angle between vectors) equals the dot product (vector multiplication). Dot products are extremely fast on numpy — one matrix multiply scores ALL chunks at once. Without normalization you'd need the full cosine formula for each pair.
class DenseRetriever(Retriever): name: str = "dense" def __init__(self, model_name=None, batch_size=64, encoder: Callable[[list[str]], np.ndarray] | None = None): self._model_name = model_name self.batch_size = batch_size self._encoder_override = encoder self._st_model: object | None = None self._chunks: list[Chunk] = [] self._vectors: np.ndarray | None = None self._indexed: bool = False
Injectable encoder — the testing pattern encoder: Callable[[list[str]], np.ndarray] | None = None — if you pass a function here, it replaces the real SentenceTransformer model. The tests use this to pass a fake fast encoder instead of downloading a 90MB model. self._st_model: object | None = None — the real model is None until first use. Why lazy loading: if you import dense.py at the top of your program, you don't want a 90MB model download to happen immediately. The model downloads only when the first real search happens. This is why offline tests are fast — they use the fake encoder and never trigger a download.
def _encode(self, texts: list[str]) -> np.ndarray: if self._encoder_override is not None: result = self._encoder_override(texts) return np.array(result, dtype=np.float32) from sentence_transformers import SentenceTransformer # lazy import if self._st_model is None: model_name = self._get_model_name() self._st_model = SentenceTransformer(model_name) embeddings = self._st_model.encode(texts, batch_size=self.batch_size, show_progress_bar=False, convert_to_numpy=True) return embeddings.astype(np.float32)
Lazy import inside the method from sentence_transformers import SentenceTransformer is INSIDE the function, not at the top of the file. This means the import only runs when _encode() is called. if self._st_model is None means the model is created once and reused — not reloaded every search. Why: loading SentenceTransformer at module level would slow every import of this file and trigger a model download. Lazy loading keeps startup fast and only pays the cost when actually needed.
def index(self, chunks: list[Chunk]) -> None: self._chunks = list(chunks) texts = [c.text for c in chunks] matrix = self._encode(texts) self._vectors = _l2_normalize(matrix) self._indexed = True
[c.text for c in chunks] — list comprehension Extracts just the text field from every Chunk object into a plain list of strings. _encode(texts) then converts all of them to a matrix of shape (n_chunks, 384) — one 384-dimensional vector per chunk. After L2 normalization, self._vectors is a matrix where every row is a unit-length vector.
def search(self, query: str, k: int = 10) -> list[ScoredChunk]: q_vec = self._encode([query]) # shape (1, 384) q_vec = _l2_normalize(q_vec) scores = (self._vectors @ q_vec.T).squeeze(axis=1) # shape (n,) actual_k = min(k, len(self._chunks)) top_indices = np.argpartition(scores, -actual_k)[-actual_k:] top_indices_sorted = sorted( top_indices.tolist(), key=lambda i: (-float(scores[i]), self._chunks[i].chunk_id), ) ...
Matrix multiply for cosine similarity self._vectors @ q_vec.T — @ is the matrix multiply operator. self._vectors is shape (n, 384), q_vec.T is shape (384, 1), result is shape (n, 1). .squeeze(axis=1) removes the trailing dimension → shape (n,). Each element is the cosine similarity between that chunk and the query. Why one matrix multiply instead of a loop: numpy executes this in optimized C/BLAS code. Scoring 1000 chunks takes microseconds. A Python for-loop over 1000 chunks doing individual dot products would take ~1000× longer.
np.argpartition(scores, -actual_k)[-actual_k:] — returns the indices of the top-k scores WITHOUT fully sorting. O(n) instead of O(n log n). Then we sort just those k indices.

8. retrieval/fusion.py — Combining BM25 + Dense

BM25 and Dense each return their own ranked list. Fusion merges them into one better list.

src/lodestone/retrieval/fusion.py
_EPSILON = 1e-9 # prevents divide-by-zero in min-max normalisation
Module-level constant 1e-9 = 0.000000001. When dividing by (max - min), if max equals min (all scores identical), you'd divide by zero. Adding epsilon prevents the crash while being small enough to not affect real scores.
class HybridRetriever(Retriever): def __init__(self, retrievers, strategy="rrf", rrf_k=None, weights=None): if not retrievers: raise ValueError("HybridRetriever requires at least one child retriever.") if strategy not in ("rrf", "weighted"): raise ValueError(f"Unknown fusion strategy '{strategy}'.") ... self._retrievers = list(retrievers) self._strategy = strategy
Input validation in __init__ Three explicit checks before storing anything. Each raises a clear error message. if not retrievers — an empty list is falsy in Python. strategy not in ("rrf", "weighted") — the tuple is the allowed set of values. Why validate in __init__: fail fast and clearly. If you skip validation, you'd get a cryptic error deep in search() when it's hard to trace back to the bad input.
@property def name(self) -> str: return "hybrid_rrf" if self._strategy == "rrf" else "hybrid_weighted"
@property — computed attribute @property makes a method callable without parentheses: retriever.name not retriever.name(). The name is computed from self._strategy dynamically instead of being stored as a fixed string. Why a property here: because the name depends on the strategy, which could theoretically change. A property keeps the name always in sync with the actual strategy.
def index(self, chunks: list[Chunk]) -> None: for r in self._retrievers: r.index(chunks) self._indexed = True
Delegating index() to children The hybrid retriever doesn't index itself — it tells each child retriever to index. for r in self._retrievers iterates over [bm25_retriever, dense_retriever] and calls index(chunks) on each. Pattern: "delegate and coordinate" — the hybrid is a coordinator, not a doer. Each child handles its own indexing format (inverted index vs numpy matrix). The hybrid just orchestrates.
def search(self, query: str, k: int = 10) -> list[ScoredChunk]: fetch_k = k * 3 per_retriever: list[list[ScoredChunk]] = [ r.search(query, k=fetch_k) for r in self._retrievers ] if self._strategy == "rrf": fused = self._fuse_rrf(per_retriever) else: fused = self._fuse_weighted(per_retriever) return fused[:k]
Fetch k×3 then trim to k Each child retriever is asked for 30 results (if k=10). This gives the fusion algorithm more material to work with — a chunk ranked #12 by BM25 but #1 by Dense might be the true best result and would be missed if each retriever only returned top 10. After fusing, trim to k.
def _fuse_rrf(self, per_retriever): rrf_scores: dict[str, float] = {} chunk_by_id: dict[str, Chunk] = {} for results in per_retriever: for rank, sc in enumerate(results, start=1): cid = sc.chunk.chunk_id rrf_scores[cid] = rrf_scores.get(cid, 0.0) + 1.0 / (self._rrf_k + rank) if cid not in chunk_by_id: chunk_by_id[cid] = sc.chunk
RRF — Reciprocal Rank Fusion enumerate(results, start=1) gives (1, first_result), (2, second_result), etc. — 1-indexed ranks. For each chunk, its RRF score += 1.0 / (60 + rank). A chunk ranked #1 by BM25 gets 1/61 ≈ 0.016. Ranked #30 gets 1/90 ≈ 0.011. A chunk ranked #1 by BOTH gets 0.016 + 0.016 = 0.032 and rises to the top. chunk_by_id deduplicates — same chunk from two retrievers is stored once. Why RRF over raw score combination: BM25 and Dense scores have completely different scales and distributions. BM25 might output 0–15, Dense 0–1. You can't add them directly. RRF uses only RANK, which is scale-agnostic. It's also been shown empirically to outperform most weighted combinations without tuning.
def _fuse_weighted(self, per_retriever): for results in per_retriever: raw_scores = [sc.score for sc in results] min_s = min(raw_scores) max_s = max(raw_scores) denom = max_s - min_s + _EPSILON norm_map = {sc.chunk.chunk_id: (sc.score - min_s) / denom for sc in results} ... for cid in chunk_by_id: total = sum(w * norm_map.get(cid, 0.0) for w, norm_map in zip(self._weights, normalised_maps)) fused_scores[cid] = total
Min-max normalization then weighted sum For each retriever's results: subtract the minimum score, divide by the range → all scores land in [0,1]. Then multiply each score by its retriever's weight and add them together. norm_map.get(cid, 0.0) gives 0 if a chunk wasn't in a retriever's results (it's penalized for not appearing).

9. retrieval/expansion.py — Query Expansion (RM3)

If you search for "cardiac arrest", you might miss documents that say "heart attack". Query expansion automatically adds related terms to your query before searching.

src/lodestone/retrieval/expansion.py
class Rm3QueryExpander: def __init__(self, retriever, fb_docs=5, fb_terms=8, orig_weight=0.6): self._retriever = retriever self.fb_docs = fb_docs self.fb_terms = fb_terms self.orig_weight = orig_weight
RM3 parameters fb_docs=5 — use the top 5 retrieved chunks as "pseudo-relevant" (assume the retriever's top results are probably correct). fb_terms=8 — add 8 new words to the query. orig_weight=0.6 — reserved for future probability weighting; currently informational only.
def expand(self, query: str) -> str: query_terms: set[str] = set(tokenize(query, remove_stops=True, stem=False)) pseudo_relevant: list[ScoredChunk] = self._retriever.search(query, k=self.fb_docs) tf: Counter[str] = Counter() for sc in pseudo_relevant: tokens = tokenize(sc.chunk.text, remove_stops=True, stem=False) tf.update(tokens) expansion_terms: list[str] = [] for term, _ in tf.most_common(): if term not in query_terms: expansion_terms.append(term) if len(expansion_terms) >= self.fb_terms: break return query + " " + " ".join(expansion_terms)
The RM3 algorithm step by step Step 1: set(tokenize(query)) — what words are already in the query? (We won't add these again.)
Step 2: self._retriever.search(query, k=fb_docs) — run the original query, get top 5 chunks.
Step 3: Counter(); tf.update(tokens) — count word frequencies across ALL 5 chunks combined.
Step 4: tf.most_common() — returns [(word, count)] sorted most-frequent first. Pick the top 8 words NOT already in the query.
Step 5: append them to the query string. Why this works: if you search "machine learning" and the top results all contain "neural", "gradient", "optimization" — those are probably relevant expansion terms. Adding them to the query improves recall on the next (real) search.
class ExpandingRetriever(Retriever): def __init__(self, inner: Retriever, expander: Rm3QueryExpander): self._inner = inner self._expander = expander @property def name(self) -> str: return f"rm3+{self._inner.name}" def search(self, query: str, k: int = 10) -> list[ScoredChunk]: expanded = self._expander.expand(query) results = self._inner.search(expanded, k=k) return [ScoredChunk(chunk=sc.chunk, score=sc.score, retriever=self.name) for sc in results]
The Wrapper / Decorator pattern ExpandingRetriever doesn't do retrieval itself — it wraps another retriever and adds a step before delegating. f"rm3+{self._inner.name}" — f-string interpolation. Result: "rm3+bm25". This pattern (wrapping an object to add behavior) is called the "Decorator pattern". Why: you can add RM3 to ANY retriever (BM25, Dense, Hybrid) without changing their code. Just wrap them. This is "composition over inheritance" — a core software design principle.

10. retrieval/rerank.py — Cross-Encoder Reranking

The hybrid retriever returns ~30 candidates. The reranker reads each one carefully alongside the query and re-scores them — picking the truly best 5–10.

Bi-encoder vs Cross-encoder:
Dense retrieval uses a bi-encoder: query and chunks are encoded separately into vectors, then compared. Fast (compare pre-computed vectors), but the query and chunk never "see" each other during encoding.

The cross-encoder encodes the (query, chunk) PAIR together. Full attention across both texts. Much more accurate, but 100× slower — so you only run it on the top ~30 candidates, not the whole corpus.
src/lodestone/retrieval/rerank.py
def _sigmoid(x: np.ndarray) -> np.ndarray: x_clipped = np.clip(x, -88.0, 88.0) return 1.0 / (1.0 + np.exp(-x_clipped))
Sigmoid function — converting logits to probabilities sigmoid(x) = 1 / (1 + e^(-x)). Converts any real number to a value between 0 and 1. Large positive x → score near 1. Large negative x → score near 0. Zero → 0.5. np.clip(x, -88, 88) prevents overflow: np.exp(89) exceeds float32 range and returns infinity. Why sigmoid: the cross-encoder outputs raw logits (unbounded numbers). Sigmoid maps them to [0,1] so scores are interpretable as approximate relevance probabilities and comparable across different queries.
class CrossEncoderReranker: def __init__(self, model_name=None, batch_size=32, scorer: Callable[[list[tuple[str,str]]], np.ndarray] | None = None): self._model_name = model_name self.batch_size = batch_size self._scorer_override = scorer self._ce_model: object | None = None
Same lazy-load + injectable scorer pattern as DenseRetriever Callable[[list[tuple[str,str]]], np.ndarray] — the type hint reads: "a callable that takes a list of (string, string) tuples and returns a numpy array". This is the signature of both the real CrossEncoder and the fake test scorer.
def rerank(self, query, candidates, top_k=5): pairs: list[tuple[str, str]] = [(query, sc.chunk.text) for sc in candidates] logits = self._score_pairs(pairs) scores = _sigmoid(logits) reranked: list[ScoredChunk] = [] for sc, new_score in zip(candidates, scores.tolist()): orig_name = sc.retriever or "unknown" reranked.append(ScoredChunk( chunk=sc.chunk, score=float(new_score), retriever=f"rerank({orig_name})", )) reranked.sort(key=lambda x: (-x.score, x.chunk.chunk_id)) return reranked[:top_k]
zip() — pairing two lists together zip(candidates, scores.tolist()) pairs each candidate with its new score: (candidate_0, score_0), (candidate_1, score_1), etc. scores.tolist() converts the numpy array to a plain Python list. f"rerank({orig_name})" updates the retriever label — so you can see in the UI that a chunk was reranked from "hybrid_rrf".

11. engine.py — The Master Coordinator

Every component so far is independent. The engine is the one file that wires everything together. The dashboard, API, and CLI all use the engine — they never import retrievers or generators directly.

src/lodestone/engine.py
class LodestoneEngine: def __init__(self, settings=None): if settings is None: from lodestone.config import get_settings settings = get_settings() self._settings = settings self._retriever = None self._reranker = None self._faithfulness_scorer = None self._loaded = False self._use_rerank = False
Lazy initialization pattern Everything is None at construction. Nothing is loaded until load() is called. This means creating an engine is instant — no models downloaded, no files read. The heavy work happens on demand. if settings is None: settings = get_settings() — lets callers inject a custom settings object (useful for tests) or use the global defaults.
def load(self, use_rerank=True): if self._loaded and self._use_rerank == use_rerank: return # idempotent from lodestone.chunking.strategies import FixedSizeChunker from lodestone.data import load_corpus from lodestone.retrieval import CrossEncoderReranker, build_default_pipeline documents = load_corpus() chunker = FixedSizeChunker() chunks = [] for doc in documents: chunks.extend(chunker.chunk(doc)) self._retriever = build_default_pipeline(chunks, settings=self._settings) if use_rerank: self._reranker = CrossEncoderReranker() self._loaded = True
Idempotent loading if self._loaded and self._use_rerank == use_rerank: return — calling load() twice does nothing on the second call. Idempotent means "calling multiple times has the same effect as calling once". chunks.extend(chunker.chunk(doc)) — extend adds all items from one list into another (vs append which adds the list itself as one item). Why: Streamlit reruns its script on every user interaction. Without idempotency, models would reload every time someone moves the k-slider. The check prevents that.
def search(self, query, k=10): self._ensure_loaded() if self._use_rerank and self._reranker is not None: candidates = retriever.search(query, k=k * 3) results = reranker.rerank(query, candidates, top_k=k) else: results = retriever.search(query, k=k) return results
Two-stage retrieval in 4 lines When reranking is ON: get 3× more candidates than needed (k*3), then rerank to pick the best k. When OFF: just get k directly. The _ensure_loaded() call triggers lazy loading if this is the first search — no manual setup required.
def ask(self, query, k=5, score_faithfulness=False): t_start = time.perf_counter() chunks = self.search(query, k=k) answerer = get_answerer(settings=self._settings) answer = answerer.answer(query, chunks) if score_faithfulness: if self._faithfulness_scorer is None: self._faithfulness_scorer = NliFaithfulnessScorer() answer = answer.model_copy(update={"faithfulness": scorer.score(answer.text, chunks)}) latency_ms = (time.perf_counter() - t_start) * 1000.0 answer = answer.model_copy(update={"latency_ms": latency_ms}) return answer
time.perf_counter() — precise timing time.perf_counter() returns the current time in fractional seconds with nanosecond precision. Taking the difference at the end gives total elapsed time. Multiply by 1000 for milliseconds.
answer.model_copy(update={...}) — immutable update Pydantic v2 models are designed to be immutable (you shouldn't change their fields after creation). model_copy(update={"faithfulness": 0.87}) creates a NEW Answer object that's identical to the original except for the faithfulness field. The original is unchanged. Why immutable: if you passed the same Answer object to multiple places and mutated it, one place's change would surprise another. Immutable objects are predictable and safe to pass around freely.
_engine_instance: LodestoneEngine | None = None def get_engine() -> LodestoneEngine: global _engine_instance if _engine_instance is None: _engine_instance = LodestoneEngine() _engine_instance.load() return _engine_instance
Module-level singleton via global variable _engine_instance is a module-level variable (lives in the module, not in a class). global _engine_instance inside the function declares that the function will write to the module-level variable (not create a local one). On first call: creates and loads the engine, stores it. On every subsequent call: returns the stored instance. Why: the API server, CLI, and dashboard all need the same loaded engine. Without a singleton, each one would create its own engine and download models three times. The global ensures one shared, loaded instance across the whole process.
